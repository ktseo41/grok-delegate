# grok 위임, 실제로 값어치가 있나? — 오케스트레이션 평가

실제 웹 리서치 과제로 두 라운드의 통제 실험을 돌려, **fable이 오케스트레이터로서 grok
워커들에게 수집을 맡기는 조합**이 정말 나은지 확인했다. 비교 대상은 같은 구조의 다른 워커
버전(fable + sonnet 워커, fable + deepseek 워커), 단독 실행 3종(fable 단독, sonnet 단독,
grok 단독), 그리고 sonnet에 상위 모델 자문 도구 advisor(fable)를 붙인 변형이다.

원본 산출물(프롬프트, 하네스, 실행별 `run.json`, 워커 출력, 블라인드 사본, 채점표)은 이
저장소에 포함하지 않고 별도의 실험 워크스페이스(비공개)에 보관한다. 이 문서는 모든 조합을
구성 이름으로 부르고 모든 수치를 옮겨 적어, 원자료 없이도 결과를 읽을 수 있게 했다.

사실상 단독 실행이라 표·차트에서 모두 sonnet 단독으로 묶었다.

advisor(fable) 실험에 대해 두 가지. 첫째, advisor 연결 자체는 강제 호출 테스트로 사전
검증했다 — 발동 0회는 배선 오류가 아니라 모델이 부르지 않기로 한 것이다. 둘째, 라운드 2의
advisor 2회 실행은 프롬프트가 딱 한 줄("필요하면 advisor에게 물어보라"는 힌트) 차이인데
힌트가 있어도 발동하지 않았다. 즉 처치가 전달되지 않았으므로 그 실행들은 sonnet 단독의
반복이고, 실행 간 점수 차이는 편차로 읽어야 한다. 같은 논리로 라운드 2의 sonnet 3회 실행(72/72, 70/72,
69/72)이 95.8–100%에 걸쳐 있다는 것이 sonnet 단독의 실행 간 변동폭 추정치다. 표와
차트에서는 세 실행을 평균(70.3/72)으로 묶었다.

두 라운드, 두 과제 유형을 통틀어 오답 셀이 하나도 없는 조합은 **grok 워커(142/142)와
deepseek 워커(141/141), 두 오케스트레이터 + 워커 조합뿐**이다. 핵심은 "grok이 똑똑하다"가
아니다 — grok 단독은 fable 단독과 나란히 최하위권이었다. 가치는 **절차의 이중화**에 있다:
저렴한 워커·별도 쿼터로 독립적으로 수집하고, 오케스트레이터가 모든 숫자를 원출처와 다시
대조하는 구조. 서로 다른 두 모델 계열(fable, grok)이 혼자 돌았을 때는 *같은* 함정 셀에서
실패했지만, 같은 구조는 그 셀들을 전부 맞혔다.

단, 이 점수들을 정확히 읽으려면: 두 워커 조합의 절차에는 "워커가 재시도 후에도 실패하면
오케스트레이터가 직접 수집한다"는 폴백이 하네스에 처음부터 명시돼 있었고, 라운드 2에서는
실제로 grok 3/12·deepseek 5/12 통화권이 그 폴백으로 채워졌다(라운드 1은 두 조합 모두
워커가 전부 수집). 그래서 각 점수는 워커 모델의 성적이 아니라 **수집 + 재검증 + 폴백이라는
절차 전체의 성적**이고, 워커 품질의 차이는 점수가 아니라 위 표의 실패율과 아래 비용 절에서
나타난다. 라운드 1의 "1차 무수집 7/12"도 같은 방식으로 읽어야 한다: 그 grok 워커들은
크래시한 게 아니다. 웹을 한 번도 조회하지 않고 **모델 기억만으로 그럴듯하고 완결돼 보이는
보고서를 쓴 채 정상 종료했다** — grok이 "중앙은행 정책금리 정도는 이미 안다"고 판단해
리서치를 건너뛴 것으로, 종료 코드·파일 크기·본문 어디에도 드러나지 않고 래퍼 사용량
트레일러의 도구 호출 목록에서만 잡혔다. "반드시 web fetch로 실제 페이지를 가져온 뒤
답하라"를 명시한 재시도로 7개 전부 회복됐고(각 5–10회 실제 조회), 이후 래퍼는 웹 호출
0회인 리서치 실행을 자동으로 실패 처리하도록 바뀌었다(웹 수집 게이트 — 아래 교훈 절
참고). 그리고 sonnet 워커 조합(라운드 1)은 검증 단계에서 오류 1셀을 걸러내지 못했고,
판단 함정형 과제에서는 아직 돌려보지 않았다.

## 과제 내용

- **라운드 1 — 조회형.** 12개 중앙은행의 현재 통화정책 설정, 필드 6개씩(정책수단명, 값,
  최근 변경일, 크기/방향, 다음 회의, 결정문 원문 인용 + URL). 공식 출처만 허용.
  12 × 6 = 72셀이 나와야 하지만, 싱가포르(MAS) 웹사이트가 실험 기간 내내 전면 점검
  중이어서 2셀은 채점이 불가능했다 — 라운드 1의 모든 점수는 분모가 70이다. 결과는
  전 조합 94–100% — 과제가 쉬워서 모두 만점 부근에 몰렸고 변별력이 없었다. 조합 간 차이는
  인용 필드에서만 났다.
- **라운드 2 — 판단 함정형.** 12개 통화권의 실질 정책금리: 정책금리(중앙은행) − 그 은행이
  **공식적으로 타겟팅하는 물가지수**의 최신 전년동월대비 상승률(통계청 발표), 소수점 둘째
  자리까지. 필드마다 함정을 심었다:
  - **지수 선택**: 공식 타겟 지수와 실무에서 자주 인용되는 지수가 다른 나라들(미국은 CPI가
    아니라 PCE, 스웨덴은 CPIF, 노르웨이는 CPI vs CPI-ATE, 일본은 종합 vs 신선식품 제외).
  - **발표 시점**: 물가 통계는 속보치(잠정)가 먼저 나오고 나중에 확정치로 바뀐다. 출처가
    "잠정"이라고 명시한 라벨을 존중하는지, 실행 며칠 전에 나온 최신 발표를 놓치지 않는지.
  - **출처 귀속**: 물가지수의 원출처는 중앙은행이 아니라 통계청이다.
  - **연쇄 채점**: 지수를 잘못 고르면 그 값으로 계산한 산술 셀도 함께 감점.

  12 × 6 = 72셀, 셀당 1점.

> 여기서 "함정"이란 일부러 틀리기 쉬운 지점을 문제에 심어 두는 것을 말한다 — 모델이 얼마나
> 아느냐가 아니라, 틀리기 쉬운 곳에서 실제로 틀리는지를 보기 위한 장치다.

## 결과를 신뢰할 수 있게 만든 장치

- **블라인드 채점.** 본 실험 리포트 5건을 `A…E`로 섞고, 매핑을 모르는 fable 채점 세션이
  채점했다. 결과 파일에 방법론 흔적(어떤 조합인지 드러나는 단서)이 없어야 한다는 요건을
  스캔으로 검증했고, 채점 세션의 파일 접근을 사후 감사해 정확히 `blind/*.md`만 읽었음을
  확인했다.
- **동일 프롬프트.** 짝지어 비교하는 조합들은 같은 프롬프트 파일을 썼다. advisor(fable)
  2회 실행은 문서화된 한 줄만 다르고 나머지는 바이트 단위로 동일하다(`diff` 보관).
- **당일 완료 + 정답 변동 통제.** 모든 실행과 채점을 같은 날 끝냈다. 실행과 채점 사이에
  새 물가 발표가 나오면 정답 자체가 바뀔 수 있어서, 발표 캘린더를 사전에 확인하고 그런
  경우 양쪽 다 정답으로 인정한다는 규칙을 미리 정해 두었다(실제로 발동할 일은 없었다).
- **예상을 미리 적어두기.** 어느 조합이 이길지, advisor(fable)가 저절로 발동할지 같은
  예상을 실행 전에 문서로 적어 두고, 결과가 나온 뒤 그 예상과 대조했다. 결과를 보고 나서
  해석을 끼워 맞추는 것을 막기 위한 장치다. 워커 실패 시의 처리(재시도 1회 → 오케스트레이터
  직접 수집)도 하네스 프롬프트에 사전 명시했다.
- **모델별 원시 토큰 보고.** 토큰은 모델별(fable / sonnet / haiku / grok / deepseek)
  원시값 그대로 보고하고, 서로 단가가 다른 모델들을 하나의 숫자로 합산하지 않았다.
  (haiku-4.5는 실험 대상 구성이 아니다 — Claude Code의 WebSearch가 가져온 페이지를
  자동으로 요약할 때 쓰는 모델이라 Claude 쪽 모든 실행에 등장한다.) Claude 쪽은
  `run.json`의 모델별 사용량, grok 쪽은 래퍼의 `[grok-usage]` 트레일러(워커별 ctxTokens,
  소요 시간, 도구 호출 수)에서 가져왔다. 여기서 래퍼란 이 저장소의 `scripts/grok-run.sh`,
  즉 모든 grok 실행이 거치는 실행 스크립트다: grok CLI는 자체 사용량 보고가 없어서 래퍼가
  매 세션 끝에 이 트레일러를 붙이고, 실행 모드별 가드레일(도구 allowlist, 아래에서 설명할
  웹 수집 게이트)도 함께 강제한다.
- **도구 사용 규율.** 모든 조합 공통: 스킬, MCP 없음; 서브에이전트는 그것이 곧 조합의
  워커 채널일 때만 사용(sonnet 워커 조합은 Agent 도구로 워커를 띄우고, grok·deepseek
  워커는 각자의 외부 CLI로 실행) — 그 위에 얹는 보조 용도로는 금지; 웹은
  WebSearch/WebFetch만(grok은 자체 `web_search`/`web_fetch`); curl 없음; 봇 차단(403)
  사이트는 도메인 한정 검색으로 대신하고 차단 우회는 하지 않음.

### 각 수치의 출처, 그리고 grok의 유의사항

- **Claude 모델**: `run.json` → `modelUsage`, 모델별·실행별. 1차 계측이라 정확하다.
  비용 열은 `run.json`의 `total_cost_usd` — API 정가로 환산한 값이며, 구독 사용자는 현금이
  아니라 구독 쿼터를 소모한다.
- **deepseek (codex CLI 경유)**: 세션별 정확한 `total_token_usage`(input / cached input /
  output / reasoning)가 codex 롤아웃 로그
  `$HOME/.codex/sessions/<YYYY>/<MM>/<DD>/rollout-*.jsonl`에 있다 — 실행 시간창에 해당하는
  세션들을 합산한다. OpenRouter 활동 내역 CSV와 대조 검증 가능(라운드 1은 CSV 실측값).
- **grok (grok CLI 경유)**: 이 CLI는 청구 기준이 되는 in/out 분리값을 로컬 어디에도
  기록하지 않는다. 세션 파일(`$HOME/.grok/sessions/<cwd-encoded>/<session-id>/`)이 노출하는
  것은 `signals.json → contextTokensUsed`(최종 컨텍스트 크기 — 래퍼 트레일러가 출력하는 값)와
  `updates.jsonl`의 이벤트별 누적치뿐이다. 콘솔의 Usage Explorer는 API 키 청구만 다루는데
  이번 실행은 SuperGrok 구독이라 항목별로 잡히지 않는다. 그래서 grok의 in/out은 세션
  데이터로 범위를 *추정*했다:
  - 하한 ≈ 최종 컨텍스트 × 입력 단가 + 추정 출력 (전부 캐시에 맞은 한 번의 패스라고 가정);
  - 상한 ≈ Σ(각 추론 스텝의 컨텍스트 크기, `updates.jsonl`의 토큰 추이 기준) × 입력 단가 +
    추정 출력 (아무것도 캐시되지 않았다고 가정);
  - 출력 토큰 ≈ (어시스턴트 + 추론 + 도구 호출 텍스트 길이) / 4.
  표의 네 값(두 라운드 × 워커/단독)은 같은 스크립트로 일괄 산출했고, 세션 식별은 각 실행의
  트레일러 ctxTokens 합계와 정확히 일치하는지로 검증했다. 범위가 넓은 이유는 캐싱 동작이
  클라이언트에서 보이지 않기 때문이고, 실제 값은 두 경계 사이에 있다. 현실적인 보조 계기판도
  있다: CLI가 매 실행마다 `$HOME/.grok/logs/unified.jsonl`에 남기는 주간 한도 소진율
  (`creditUsagePercent`) — 라운드 2의 grok 단독 실행은 SuperGrok 주간 쿼터를 약 1퍼센트
  포인트 움직였다. 구독 기준으로는 달러보다 이 표현이 실체에 가깝다.

차트는 `assets/gen_charts.py`로 재생성한다.

## 평가가 가르쳐준 것 (그리고 이 저장소에서 바뀐 것)

1. **위험한 실패는 크래시가 아니라 조용한 무수집이다.** 라운드 2에서 grok 워커 12개 중
   4개가 종료 코드 0으로, 웹 호출이 전혀 없이 **모델 기억만으로 작성한, 그럴듯하고 크기도
   정상인 출력**을 냈다 — 종료 코드로도, 파일 크기로도, 본문으로도 드러나지 않았다. 유일한
   신호는 사용량 트레일러의 도구 호출 목록이었고, 재시도로 회복된 것은 4개 중 1개뿐이었다.
   (라운드 1에서는 12개 중 7개가 같은 방식으로 실패했지만 재시도로 전원 회복.) 이제 래퍼는
   이것을 **웹 수집 게이트**로 강제한다: 웹 도구 호출이 하나도 없는 `research`/`research-rw`
   실행은 비정상 종료한다(`FAILED: … no web tool call`). `scripts/grok-run.sh` 참고;
   `evals/stub-regression.sh`(H6)에서 회귀 테스트됨.
2. **강한 모델의 단독 실행이 놓치는 것은 추론이 아니라 기계적 꼼꼼함이다.** 라운드 2의
   승부처는 "출처가 스스로 붙인 *잠정* 라벨을 존중하라"와 "이틀 전에 나온 최신 발표를
   스캔하라"였다. fable 단독과 grok 단독 모두 정확히 이 지점에서 놓쳤고, 평가 전체에서
   산술 오류는 하나도 없었다. 과제에 이런 신선도/라벨 함정이 있다면, 더 큰 모델이 아니라
   절차(수집 → 재검증)를 사야 한다.
3. **워커 분할은 턴 예산에서도 이긴다.** grok 세션 하나가 12개 주제를 혼자 처리하려다
   기본값 `--max-turns 30`을 넘겨 도중에 죽었다(라운드 1 grok 단독의 첫 시도). 주제별
   워커는 각각 3–17회의 도구 호출로, 워커 한 개 분량의 시간 안에 전부 끝났다. 이제 래퍼는
   실제 적용되는 턴 한도를 로그로 남기고, SKILL.md에 턴 예산 산정 규칙을 문서화했다.
4. **가만히 있는 advisor는 안전망이 아니다.** sonnet에 노출된 advisor(fable)는 두 라운드에
   걸친 4개 실행에서 **한 번도 발동하지 않았다** — 중립적인 한 줄 힌트를 줘도, sonnet이
   공식 문구를 정확히 받아 적어 놓고 엉뚱한 지수를 고른 셀에서도. 연결 자체는 강제 호출
   테스트로 검증했으므로, 부르지 않은 것은 모델의 선택이다. advisor를 발동시키려면 지시를
   강하게 써야 하는데, 그 시점부터는 "자문할 줄 아는가"가 아니라 "지시를 따르는가"를
   측정하게 된다.
5. **측정 자체가 행동을 오염시킬 수 있다.** 자식 세션에게 "advisor가 쓸 수 있는 상태인지
   기록하라"고 지시했더니 advisor에 시험 호출을 하는 부작용이 생겼다(사전 점검에서 발견,
   "도구 목록만 관찰하고 호출은 금지"로 고쳐서 해결). 계측용 문구는 미리 확정하고 본 실행
   전에 점검하라.
6. **돈이 어디로 갔는가.** 라운드 2의 fable + grok 워커 조합: 오케스트레이터(fable)는
   검증과 종합에 토큰을 썼고, grok 워커 16개(12 + 재시도 4)는 xAI 계량기에서 615k
   ctxTokens를 소모했다. 이 조합은 수집이 쿼터 밖으로 밀려난 덕분에 Claude 쪽 haiku/웹
   사용량도 전 조합 중 가장 낮았다. 위임은 과제의 무겁고 병렬화 가능한 부분을 별도 지갑으로
   옮기면서도 정확도를 잃지 않았다 — 이것과 2번 발견이 이 스킬의 존재 이유다.
7. **워커 품질은 정확도가 아니라 비용을 좌우하는 손잡이다.** 두 워커 조합 모두 "재시도
   후에도 실패하면 오케스트레이터가 직접 수집"이 사전 규칙이었고, 라운드 2에서 그 폴백이
   grok은 3/12, deepseek은 5/12 통화권에서 발동했다. 정확도는 양쪽 다 100%로 같았지만
   비용이 갈렸다: deepseek 조합은 Claude 쪽 $9.81(grok 조합 $7.66 대비 +28%), fable 출력
   토큰 전 조합 최대. 즉 구조는 오케스트레이터의 노력을 더 써서 워커 실패를 흡수하고, 더
   나은 워커를 쓰면 더 많은 작업이 저렴한 계량기에 남는다. 워커는 벤치마크 지능이 아니라
   (실패율 × 외부 단가)로 골라야 한다 — 그리고 점수만 보면 이 차이가 안 보이므로, 실패율을
   점수 옆에 함께 보고해야 한다(위 요약 표처럼).

## 다음 모델/채널에 이 틀 재사용하기

새 위임 대상(다른 CLI, 다른 모델 계열, 새 모드)을 이 결과와 비교하려면, 틀은 유지하고
조합만 바꾼다:

1. **변별하고 싶은 것에 맞춰 과제 유형을 고른다.** 단순 조회형은 모든 모델이 만점 부근에
   몰려 변별력이 없다(라운드 1). 점수 차이를 보고 싶으면 판단 함정(공식 정의 vs 통용 정의,
   발표 시점, 출처 귀속, 연쇄 채점)을 심어라. 함정의 정답은 실행 당일 다시 확인해야 한다 —
   발표 캘린더에 따라 바뀐다.
2. **레퍼런스 조합 3개를 항상 함께 돌린다**: 후보 구조(오케스트레이터 + 위임 워커),
   오케스트레이터 모델 단독, 위임 모델 단독. *구조가 두 단독 모두를 이기는가*가 발견이다 —
   후보와 단독 하나만 비교하면 모델 효과와 절차 효과가 뒤섞인다.
3. **채점 전에 결과를 블라인드로 섞고, 결과 파일에 방법론 흔적을 금지하고, 채점 세션의
   파일 접근을 사후 감사한다.** 채점자는 원출처의 원문과 대조해야 하며, 자기 배경지식으로
   셀을 채점해서는 안 된다.
4. **토큰은 모델별로, 각 채널의 1차 계량기에서 원시값으로 보고한다** — 모델 간 합산은 하지
   않는다. 외부 위임 대상이 있다면 그쪽 자체 계량기를 확보한다(이 저장소의 래퍼가
   `[grok-usage]` 트레일러를 출력하는 이유다).
5. **워커 출력은 종료 코드가 아니라 수집 증거(도구 호출 횟수/목록)로 게이트한다.** 재시도
   예산과 그 이후의 폴백(오케스트레이터 직접 수집 허용 여부)을 실행 전에 하네스에 명시하고,
   **폴백이 발동한 횟수를 점수 옆에 반드시 함께 보고한다** — 점수만으로는 워커가 일을 한
   조합과 오케스트레이터가 메꾼 조합이 구분되지 않는다.
6. **실행 순서는 저렴한 것부터, 모든 조합과 채점을 하루 안에 끝내고, CLI와 모델 버전을
   기록한다** (이번 실행: claude CLI 2.1.206, grok 0.2.93, grok-4.5, claude-sonnet-5 /
   claude-fable-5).

## 남은 유의사항과 후속 과제

- **조합당 1회 실행.** 라운드 2의 sonnet 3회 실행이 3셀 범위(95.8–100%)로 흩어져 있으므로,
  단독 실행끼리의 최상위 동률(sonnet 단독 vs grok 워커 조합)은 편차 범위 안이다. 1회
  실행에서도 살아남는 것은 *연속 무결점*(grok 워커 조합의 두 라운드 142/142)과 단독
  실행들이 *같은 함정에서 같은 방식으로 실패했다는 지문*이다.
- **사후 실행 조합(라운드 2의 grok 단독, fable + deepseek 워커)은 블라인드가 아니라 확정
  정답표와 대조해 채점했다** — 서로 간에는 방법이 같아 비교 가능하고, 블라인드 채점된 본
  실험과도 방향성 있는 비교는 가능하다. 라운드 1의 deepseek 워커 조합은 완전한 블라인드
  조합이었다.
- **후속 실험 후보.** ① fable + sonnet 워커를 판단 함정형 과제(라운드 2 유형)로 재검증 —
  라운드 1에서 이 조합만 검증 단계에서 오류 1셀을 놓쳤는데, 함정형 과제에서도 그런지 확인이
  필요하다. ② haiku 워커 조합 — "워커는 실패율 × 단가로 고른다"는 결론에 데이터 포인트를
  하나 더 보탠다. ③ 구조를 분해해서 결과를 만든 게 어느 부분인지 확인 — 현 설계로는
  "재검증 워크플로우가 있다"와 "워커에게 위임한다"를 분리할 수 없다. 세 가지 탐침:
  (a) fable + grok 워커에서 재검증 패스를 *뺀* 조합(위임만 — 독립 수집만으로도 버티는가);
  (b) 단독 모델에 같은 수집 → 재검증 워크플로우를 프롬프트로 지시한 조합(워크플로우만 —
  단, 자기 답을 자기가 재검증하면 그 답에 앵커링될 위험이 있다); (c) 교차 모델 검증 —
  한 모델이 수집하고 다른 모델 계열이 재검증 — (b)의 앵커링 위험을 통제하는 변형.
- **grok 0.2.93의 `research` 모드는 실패로 잠긴다(fail closed)** — 웹 도구와 읽기 전용
  allowlist를 함께 쓸 때 나는 업스트림 버그. 그래서 이번 평가 워커들은 사용자의 명시적
  동의를 받아 `research-rw`로 실행했다. xAI가 버그를 고치면 같은 틀로 `research`(읽기 전용)
  워커를 직접 비교할 수 있다.
